# 翻译（i18n）与文本域约定

AIYA-CMS 主题的翻译分为两条链路：

- PHP 侧（主题与自带插件）：使用 WordPress 的 gettext 体系（`.pot/.po/.mo`）
- 前端（Vite + React）：使用 `@wordpress/i18n`，由 WordPress 把翻译编译为 JSON 并在前端注入

## 文本域（Text Domain）

本仓库约定只使用两个文本域：

### `aiya-cms`（主题域）

- 适用范围：主题本体（`functions.php`、`inc/`、`templates/`）与前端（`src/`）
- PHP 示例：
  - `__('搜索', 'aiya-cms')`
- 前端示例：

```ts
import { __, sprintf } from '@wordpress/i18n';

__('搜索', 'aiya-cms');
sprintf(__('欢迎，%s', 'aiya-cms'), userName);
```

### `aiya-framework`（插件域）

- 适用范围：仅限仓库内的 `plugins/` 目录（自带框架/插件）
- PHP 示例：
  - `__('搜索', 'aiya-framework')`

约定：`plugins/` 内的字符串不要使用 `aiya-cms`，避免主题与插件翻译相互污染。

## 前端翻译加载机制（概览）

前端 bundle 使用 Vite 以 ESM 方式注入（不是 `wp_enqueue_script()`），因此需要显式把 WordPress 注入到页面的 localeData 同步到前端的 `@wordpress/i18n` 实例。

- PHP 侧通过 `wp_set_script_translations()` 让 WordPress 在页面里加载 `languages/*.json`
- 前端在启动阶段调用 `syncWordPressTranslations()`（见 `src/lib/utils.ts`）把 `window.wp.i18n` 的 localeData 写入 `@wordpress/i18n`

## 更新翻译文件的流程

### 1) 更新模板（POT）

更新/生成翻译模板：

- `composer run i18n:pot`：同时生成主题与插件的 `.pot`

产物：

- `languages/aiya-cms.pot`
- `plugins/languages/aiya-framework.pot`

### 2) 维护语言文件（PO/MO）

使用 Poedit 或类似工具把 `.pot` 的变更同步到对应 locale 的 `.po`，并生成 `.mo`。

约定：

- `aiya-cms` 的 `.po/.mo` 放在 `languages/`
- `aiya-framework` 的 `.po/.mo` 放在 `plugins/languages/`

### 3) 生成前端 JSON（给 `wp_set_script_translations` 使用）

- `composer run i18n:json`

说明：

- 该命令会把现有的 `.po/.mo` 编译为前端可加载的 JSON
- JSON 文件默认会放在 `languages/` 下（`--no-purge` 不会清理旧 JSON）

## 常见约束

- 所有新增 UI 文案（前端）必须走 `@wordpress/i18n`，并且 domain 固定为 `aiya-cms`
- `plugins/` 目录的 PHP 文案必须使用 `aiya-framework`
